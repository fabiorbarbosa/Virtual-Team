# Complexity Guard Skill

## Mission
Prevent overengineering and keep architecture proportional to validated requirements and team maturity.

## Use when
- Advanced or distributed patterns are proposed.
- New abstraction layers are being introduced.
- Scale assumptions are uncertain.

## Required inputs
- Problem statement
- Hard non-functional requirements
- Current scale metrics
- Team expertise
- Simpler alternatives

## Checklist
- Is complexity driven by current needs?
- Were simpler options proven insufficient?
- What operational burden is added?
- What new failure modes appear?
- Can the team build and operate this safely?
- Is there an exit strategy?

## Deliverables
- Complexity assessment
- Minimal viable architecture recommendation
- Criteria for when extra complexity would be justified later
- Guardrails against architecture drift
